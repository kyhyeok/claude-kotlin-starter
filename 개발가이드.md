# 개발 가이드

## 1. 아키텍처

**헥사고날 아키텍처 + 도메인 모델 패턴**.

```
[외부 Actor] → [adapter] → [application] → [domain]
```

의존 방향은 ArchUnit `ArchitectureTest`가 강제한다. 이 테스트가 깨지면 PR이 머지될 수 없다.

### 계층별 책임

| 계층 | 의존 가능 | 의존 불가 | 책임 |
|---|---|---|---|
| `domain` | (없음, JPA만 허용) | Spring, Adapter | 도메인 객체, VO, 도메인 서비스 |
| `application` | domain | Adapter | Use Case 인터페이스 + 구현, 트랜잭션 경계 |
| `adapter` | application, domain | 없음(이미 외곽) | HTTP, JPA, Redis, JWT 등 외부 통신 |
| `config` | 모두 | — | 횡단 Bean (Clock, Jackson 등) |

### 패키지 명명 규칙

- `application/provided/` — "내가 제공" (use case 인터페이스). 컨트롤러는 이 인터페이스만 의존.
- `application/required/` — "내가 요구" (포트). Repository, JwtIssuer 등.
- `adapter/<technology>/` — 기술별 어댑터.

## 2. 도메인 모델 작성 규칙

```kotlin
// ✅ 올바른 예
@Entity
class Member private constructor(
    @Id @GeneratedValue val id: Long? = null,
    val email: Email,        // VO
    private var status: MemberStatus,
    private var passwordHash: String,
) {
    companion object {
        fun register(req: RegisterRequest, encoder: PasswordEncoder): Member { ... }
    }

    fun activate() {
        check(status == PENDING) { "PENDING 상태가 아닙니다" }
        status = ACTIVE
    }
}
```

- 정적 팩토리(`Companion.register()`)로 생성 책임을 도메인에 둔다.
- `var` 노출 금지 → 행위 메서드(`activate()`)로 상태 전이.
- 상태 가드는 `check(...)`로 도메인 안에서.

## 3. Value Object

- `record` 대신 `data class` 또는 `@JvmInline value class`.
- 생성자에서 invariant 검증.

```kotlin
@JvmInline
value class Email(val address: String) {
    init {
        require(EMAIL_REGEX.matches(address)) { "이메일 형식이 바르지 않습니다: $address" }
    }
    companion object {
        private val EMAIL_REGEX = Regex("...")
    }
}
```

## 4. 시간

`LocalDateTime.now()` 직접 호출 금지. `Clock` Bean 주입.

```kotlin
class PaymentService(private val clock: Clock) {
    fun prepare() = Payment.createPrepared(now = LocalDateTime.now(clock))
}
```

## 5. 테스트 컨벤션

### 5.1 명명: 백틱 한국어
```kotlin
@Test fun `200을 반환한다`() { ... }
```

### 5.2 디렉토리 = API URL
```
test/.../api/health/GET_specs.kt
test/.../api/auth/login/POST_specs.kt
```

### 5.3 메타 어노테이션
```kotlin
@ApplicationApiTest
@DisplayName("GET /health")
class `GET_specs` { ... }
```

### 5.4 테스트 데이터
- `support/generator/`의 `EmailGenerator`, `UsernameGenerator` 사용 → 매 테스트가 독립된 데이터.
- 하드코딩 PK (`findById(2L)`) 사용 금지.

### 5.5 단언
- self-comparison(`assertThat(x.a).isEqualTo(x.a)`) 금지 — 항진식.
- 도메인 특화 assertion이 반복되면 `support/assertion/`에 추출.

## 6. 금지 사항 (실제 사례 기반)

흔히 반복되는 결함 패턴을 박제:

### 6.1 시크릿을 yml에 평문 저장 ❌
- 환경변수(`${JWT_SECRET}`) + Vault/Secrets Manager 사용.
- ADR-0004 참고.

### 6.2 DTO에 JPA 엔티티 필드 노출 ❌
- 권한 우회 면이 됨. VO + path/principal에서 추출.

### 6.3 인증 필터가 매 요청 DB 조회 ❌
- 토큰 클레임으로 PrincipalDetails 구성 또는 캐싱.
- starter는 OAuth2 Resource Server 사용 → 자동 해결.

### 6.4 JpaRepository를 controller가 직접 의존 ❌
- 영속성 모델 누수. `application/required/` 포트만 의존.

### 6.5 컨트롤러에서 ServiceImpl 직참조 ❌
- DIP 위배. 인터페이스만 의존.

### 6.6 `ddl-auto: create/update` ❌
- 스키마는 Flyway가 단일 책임. JPA는 `validate`만.

## 7. 새 도메인 추가 절차

1. `domain/<name>/` 폴더 생성, 도메인 객체 + VO 작성.
2. `application/<name>/provided/` Use Case 인터페이스 정의.
3. `application/<name>/required/` 포트 인터페이스 정의 (필요 시).
4. `application/<name>/<Name>Service.kt` 구현.
5. `adapter/persistence/<Name>JpaRepository.kt` 등 어댑터 구현.
6. `adapter/webapi/<Name>Api.kt` 컨트롤러.
7. `test/api/<name>/<METHOD>_specs.kt` 테스트 작성.

ArchUnit + ktlint + detekt가 통과하면 PR 가능.
