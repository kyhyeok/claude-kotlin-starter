# ADR-0009: DB 스키마는 사람이 단일 책임자, LLM은 read-only

- 상태: Accepted
- 일시: 2026-05-07

## 맥락

LLM이 자유롭게 마이그레이션 파일을 작성·수정할 수 있으면 다음 위험이 누적된다:

- 스키마 변경의 의사결정 주체가 흐려진다 — "누가 왜 이 컬럼을 추가했는가"의 추적이 불가.
- LLM이 만든 잘못된 마이그레이션이 production DB에 적용될 위험. 데이터 손실/이행 실패의 책임 소재가 불분명.
- 변경 이력 감사가 어려워진다 — LLM 변경과 사람 변경이 git history에서 구분되지 않음.
- starter-kit의 기존 운영 정책(ADR-0001 헥사고날 + `ddl-auto: validate` + Flyway 단일 책임자)의 일관된 연장선이 필요.

## 결정

**DB 스키마(`src/main/resources/db/migration/V*.sql`)는 사람이 단일 책임자.**

- LLM은 `db/migration/` 디렉토리의 파일을 **작성·수정·삭제하지 않는다**.
- LLM은 기존 마이그레이션 파일을 **읽기만** 한다 — JPA 엔티티, Repository, jOOQ 매핑 등 후속 코드의 입력으로 사용.
- 스키마 변경이 필요하면 LLM은 **권장 SQL만 텍스트로 제안**한다. 실제 파일 작성은 사람이 한다.
- DB에 직접 영향을 주는 명령(`./gradlew flywayMigrate`, `psql`, `./gradlew bootRun`, 임의 SQL 실행)은 LLM이 실행하지 않는다 — 모두 사람이 실행.

## 결과

- 작업 분담 명확: **사람 = 스키마**, **LLM = 스키마를 따르는 코드**.
- production 사고 위험 차단. LLM의 마이그레이션 자동 작성으로 인한 의도치 않은 컬럼 변경/제거 불가.
- 새 프로젝트 fork 시에도 동일 정책 자동 적용 (CLAUDE.md §6 + 본 ADR).
- LLM은 jOOQ codegen 결과/JPA 매핑/Repository를 작성할 때 항상 현재 스키마를 우선 읽고 정렬한다.

## 거부된 대안

- **LLM이 자유롭게 마이그레이션 작성**: 빠르지만 변경 이력 감사·책임 소재 불분명. production 사고 위험.
- **PR 리뷰만으로 통제**: PR 단위 리뷰는 LLM이 만든 다수 변경에 묻혀 누락 가능. 강제 분리가 더 안전.
- **사람이 LLM의 마이그레이션 제안을 검토 후 commit**: 검토 자체에 LLM 신뢰가 들어감 — 분리가 강제되지 않음.
