# ADR-0009: DB 스키마 적용은 사람 단일 책임자, 파일 작성은 LLM 가능

- 상태: Accepted
- 일시: 2026-05-07
- 갱신: 2026-05-07 — 분담 라인 정밀화. 파일 작성은 LLM 가능으로 완화, DB 적용 명령만 사람 전용으로 좁힘.

## 맥락

LLM이 자유롭게 DB에 변경을 적용할 수 있으면 다음 위험이 누적된다:

- 사람의 의도와 다른 시점에 DB 스키마가 물리적으로 변경될 수 있다.
- production 사고/데이터 손실 시 책임 소재가 불분명.
- 변경 적용의 통제권이 사람에게 있어야 staging→production 흐름이 안전.

반면 마이그레이션 파일(`db/migration/V*.sql`)의 **작성** 자체는 git diff로 명시적이고 사람 검토 가능 — 파일 작성과 DB 적용을 분리하면 LLM의 작업 속도를 살리면서 적용 통제권은 사람에게 둘 수 있다.

starter-kit의 기존 운영 정책(ADR-0001 헥사고날 + `ddl-auto: validate` + Flyway 단일 책임자)의 일관된 연장선.

## 결정

**DB 변경의 물리적 적용은 사람이 단일 책임자. 스키마 파일 작성은 LLM이 가능하다.**

| 영역 | LLM | 사람 |
|---|---|---|
| `db/migration/V*.sql` 파일 작성·수정 | ✓ | 검토·승인 |
| 마이그레이션 SQL 결정 (컬럼/인덱스/제약) | 권고 | 최종 결정 |
| `./gradlew bootRun` (Flyway 자동 적용 시점) | ✗ | ✓ |
| `./gradlew flywayMigrate` | ✗ | ✓ |
| `psql`, 임의 SQL 실행 | ✗ | ✓ |
| **이미 적용된** V*.sql 파일 삭제·수정 | ✗ | ✓ |

핵심: **물리적 DB 변경 시점은 항상 사람이 통제**한다.

## 결과

- LLM이 마이그레이션 파일을 작성해 작업 속도 확보. 사람은 git diff로 검토.
- DB 변경 적용은 사람이 단일 통제 — production 사고 위험 차단.
- 이미 적용된 마이그레이션 파일은 LLM이 손대지 않음 — git history 무결성과 운영 DB와의 정합성 보장.
- 새 프로젝트 fork 시에도 동일 정책 적용 (CLAUDE.md §6 + 본 ADR).

## 거부된 대안

- **LLM이 모든 DB 작업 자유**: production 사고 위험. 적용 시점 통제 불가.
- **LLM은 마이그레이션 파일도 작성 금지** (본 ADR의 초기 결정): 너무 보수적. 파일 변경은 git diff로 검토 가능하므로 적용 통제만 분리하면 충분. LLM이 매번 SQL을 텍스트로만 제안하는 것은 비효율.
- **PR 리뷰만으로 통제**: 적용 통제와 분리되지 않음. LLM이 직접 bootRun 실행하면 사람 review 전에 DB가 변경됨.
