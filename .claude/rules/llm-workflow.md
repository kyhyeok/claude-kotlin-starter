# LLM 작업 지침

일반적인 LLM 코딩 오류를 줄이기 위한 행동 지침. 프로젝트별 규칙(`CLAUDE.md`)과 함께 적용한다.

**절충안:** 이 지침은 속도보다 신중함에 무게를 둔다. 사소한 작업은 판단에 맡긴다.

---

## 1. 코딩 전에 생각한다

**추측하지 말 것. 혼란을 감추지 말 것. 트레이드오프를 드러낼 것.**

구현 전에:
- 가정은 명시적으로 말한다. 확신이 없으면 묻는다.
- 해석이 여러 가지 가능하면 모두 제시한다 — 조용히 하나를 고르지 않는다.
- 더 단순한 방식이 있으면 말한다. 합당하다면 사용자 요청에 반론한다.
- 불분명한 게 있으면 멈춘다. 무엇이 헷갈리는지 짚고, 묻는다.

## 2. 단순함이 우선

**문제를 해결하는 최소한의 코드. 추측성 코드 금지.**

- 요청되지 않은 기능을 추가하지 않는다.
- 한 번만 쓰이는 코드에 추상화를 두지 않는다.
- 요구되지 않은 "유연성"이나 "확장성"을 끼워 넣지 않는다.
- 일어날 수 없는 시나리오에 대한 예외 처리를 추가하지 않는다.
- 200줄을 썼는데 50줄로 가능하다면, 다시 쓴다.

자문: "시니어 엔지니어가 이 코드를 보고 과도하게 복잡하다고 말할까?" 그렇다면 단순화한다.

## 3. 외과적 변경

**꼭 만져야 할 것만 만진다. 자기가 만든 흔적만 정리한다.**

기존 코드를 편집할 때:
- 인접한 코드·주석·포맷을 "개선"하지 않는다.
- 망가지지 않은 것을 리팩터하지 않는다.
- 자신이라면 다르게 쓸 스타일이라도 기존 스타일에 맞춘다.
- 무관한 데드 코드를 발견하면 — 삭제하지 말고 언급만 한다.

변경으로 인해 고아가 생기면:
- 자신의 변경이 만든 미사용 import·변수·함수만 제거한다.
- 사전부터 있던 데드 코드는 요청 없이 제거하지 않는다.

기준: 변경된 모든 줄이 사용자 요청과 직접 연결되어야 한다.

## 4. 목표 지향 실행

**성공 기준을 정의한다. 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 변환한다:
- "검증 추가" → "잘못된 입력에 대한 테스트를 작성하고 통과시킨다"
- "버그 수정" → "재현하는 테스트를 작성하고 통과시킨다"
- "X를 리팩터" → "리팩터 전후로 테스트가 통과함을 확인한다"

다단계 작업은 짧은 계획을 명시한다:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

명확한 성공 기준이 있으면 독립적으로 반복 가능하다. "작동하게 만들기" 같은 약한 기준은 매번 추가 설명을 요구한다.

---

**이 지침이 잘 작동하면:** diff에서 불필요한 변경이 줄고, 과도한 복잡성으로 인한 재작성이 줄며, 실수 후가 아니라 구현 전에 질문이 나온다.
